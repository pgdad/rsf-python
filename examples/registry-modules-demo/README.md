# Registry Modules Demo

This example deploys an RSF image-processing workflow as a Lambda Durable Function
using Terraform registry modules (`terraform-aws-modules/lambda`) via RSF's custom
provider system. It demonstrates RSF's IaC-agnostic architecture — use community
Terraform modules instead of RSF's built-in Terraform generation.

## Architecture

```
rsf deploy workflow.yaml
    └── CustomProvider (reads rsf.toml)
            └── deploy.sh deploy
                    ├── zip src/generated/ + handlers/ → build/function.zip
                    └── terraform apply
                            └── Lambda Durable Function + IAM role
                                    └── aws_lambda_alias "live"  ← invoke this ARN
```

## Prerequisites

- Python 3.11+
- RSF installed: `pip install -e .` (from project root)
- Terraform >= 1.5.7
- AWS CLI configured (profile: `adfs`, region: `us-east-2`)
- `jq` installed (`apt install jq` / `brew install jq`)

## Quick Start

**1. Install RSF:** `pip install -e .` from the project root.

**2. Set the absolute path in rsf.toml:**

```bash
echo "$(cd examples/registry-modules-demo && pwd)/deploy.sh"
```

Paste the printed path into `rsf.toml` replacing the `program = "/REPLACE/..."` placeholder.

**3. Ensure deploy.sh is executable:**

```bash
chmod +x examples/registry-modules-demo/deploy.sh
```

**4. Deploy:**

```bash
cd examples/registry-modules-demo
rsf deploy workflow.yaml
```

RSF generates orchestrator code into `src/generated/`, then invokes `deploy.sh deploy`
which zips the source and runs `terraform apply`. The alias ARN and a sample
`aws lambda invoke` command are printed after a successful deploy.

**5. Invoke the function:**

```bash
aws lambda invoke \
  --function-name '<alias-arn-printed-above>' \
  --payload '{"image_url": "s3://my-bucket/photo.jpg"}' \
  --cli-binary-format raw-in-base64-out \
  response.json && cat response.json
```

**6. Teardown:**

```bash
rsf deploy --teardown workflow.yaml
```

## Directory Structure

```
registry-modules-demo/
  workflow.yaml          # Image-processing pipeline definition
  rsf.toml               # Custom provider config (set program path here)
  deploy.sh              # Invoked by RSF: zip source + terraform apply/destroy
  handlers/              # One handler file per Task state
    validate_image.py
    resize_image.py
    analyze_content.py
    catalogue_image.py
  terraform/             # Terraform configuration (Lambda + IAM)
    versions.tf          # Pinned provider versions
    main.tf              # Lambda module + alias "live"
    iam_durable.tf       # Managed policy + inline supplement documentation
    variables.tf         # workflow_name, execution_timeout, aws_region
    outputs.tf           # alias_arn, function_name, role_arn
    backend.tf           # Local backend (state gitignored)
  build/                 # Generated at deploy time (gitignored)
    function.zip
  src/generated/         # Generated by rsf deploy (gitignored)
  tests/
    test_handlers.py     # 16 unit tests across all 4 handlers
```

## Workflow Description

The `image-processing` pipeline consists of 4 Task states:

1. **ValidateImage** — checks format and file size; raises `InvalidImageError` on failure
2. **ResizeImage** — computes target dimensions while preserving aspect ratio
3. **AnalyzeContent** — derives content tags from image format
4. **CatalogueImage** — writes a catalogue entry with ISO timestamp

Includes Retry/Catch blocks, a DynamoDB table declaration (`image-catalogue`), and a
Dead Letter Queue (`image-processing-dlq`). The DynamoDB/DLQ Terraform is added in
Phase 58 — the declarations in `workflow.yaml` are for forward compatibility.

## How It Works

1. `rsf deploy workflow.yaml` loads the workflow, runs code generation into `src/generated/`
2. CustomProvider reads `rsf.toml`, sets `RSF_METADATA_FILE` via FileTransport, invokes `deploy.sh deploy`
3. `deploy.sh` reads metadata (workflow name, timeout) via `jq`, zips `src/generated/` + `handlers/` into `build/function.zip`
4. `terraform apply` provisions the Lambda function + IAM role + `aws_lambda_alias "live"`
5. The alias ARN is printed — always invoke via alias, never `$LATEST` (AWS issue #45800)
