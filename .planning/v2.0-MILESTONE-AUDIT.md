## Integration Check Results

### Cross-Phase Wiring Issues

**1. ChaosFixture / rsf test gap (TEST-02)**
- `ChaosFixture.patch()` requires a `MockDurableContext` with a `.step()` method
- `LocalRunner` in `test_cmd.py` calls handlers directly via `_call_handler_with_retry()`
- No bridge between the two surfaces — `rsf test --chaos` does not exist
- **Impact:** Users cannot inject chaos failures via the CLI test command; must use `MockDurableContext` separately

**2. GitHub Action workflow-file forwarding (ECO-03)**
- `action/entrypoint.sh` passes `${WORKFLOW_FILE}` to `rsf validate` and `rsf generate`
- `rsf deploy` is called without `${WORKFLOW_FILE}` — defaults to `workflow.yaml`
- **Impact:** Breaks when `workflow-file` input is a non-default path with `deploy: true`

### Design Notes (Non-Blocking)

- **Schema sync:** VS Code extension bundles a static copy of `rsf-workflow.json` (byte-identical to canonical). No automated sync mechanism exists.
- **Semantic validator parity:** TypeScript port in VS Code matches all 7 Python validation categories exactly — string-for-string error messages.
- **Template simplicity:** Both templates use basic ASL only — no v2.0 DSL features demonstrated in template `workflow.yaml` files.
- **OBS-03 design:** CloudWatch metrics are intentionally example-only, not generated by template.
