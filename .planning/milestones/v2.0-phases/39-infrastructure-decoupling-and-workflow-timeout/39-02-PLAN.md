  <behavior>
    DSL Parsing:
    - Input: `{"StartAt": "T", "TimeoutSeconds": 300, "States": {"T": {"Type": "Task", "End": true}}}` -> parsed successfully, definition.timeout_seconds == 300
    - Input: `{"StartAt": "T", "TimeoutSeconds": 0, "States": {"T": {"Type": "Task", "End": true}}}` -> Pydantic validation error (0 is not valid)
    - Input: `{"StartAt": "T", "TimeoutSeconds": -5, "States": {"T": {"Type": "Task", "End": true}}}` -> Pydantic validation error (negative)
    - Input: `{"StartAt": "T", "States": {"T": {"Type": "Task", "End": true}}}` -> parsed successfully, definition.timeout_seconds is None (optional)

    Semantic Validation:
    - Workflow with `TimeoutSeconds: 300` -> no validation errors
    - Workflow with `TimeoutSeconds: 99999999` -> warning (extremely large timeout, >30 days)
    - Workflow without timeout -> no validation errors

    Code Generation:
    - When timeout_seconds is set: orchestrator.py contains `import time`, a `_workflow_start_time = time.monotonic()` before the while-loop, and a `WorkflowTimeoutError` check at the top of each while-loop iteration
    - When timeout_seconds is None: orchestrator.py does NOT contain any timeout-related code (no import time, no start_time, no timeout check) — backward compatible
    - The WorkflowTimeoutError class is always present in the template (alongside WorkflowError) for consistency
  </behavior>
  <implementation>
    1. **DSL Model** (`src/rsf/dsl/models.py`):
       - Change `timeout_seconds` field from `ge=0` to `gt=0` (strictly positive). This makes Pydantic reject 0 and negative values at parse time.
       - This is a one-line change: `ge=0` -> `gt=0`.

    2. **Semantic Validator** (`src/rsf/dsl/validator.py`):
       - Add a `_validate_timeout` function called from `validate_definition`.
       - If `definition.timeout_seconds` is not None and `definition.timeout_seconds > 2592000` (30 days in seconds), add a warning: "TimeoutSeconds value {n} exceeds 30 days — this is unusually large".
       - This is a warning (severity="warning"), not an error.

    3. **Code Generator** (`src/rsf/codegen/generator.py`):
       - In `render_orchestrator`, pass `timeout_seconds=definition.timeout_seconds` to the template context.

    4. **Orchestrator Template** (`src/rsf/codegen/templates/orchestrator.py.j2`):
       - Add `WorkflowTimeoutError` class (sibling to `WorkflowError`):
         ```python
         class WorkflowTimeoutError(Exception):
             """Raised when the workflow exceeds its configured timeout."""
             def __init__(self, timeout_seconds: int, elapsed: float):
                 self.timeout_seconds = timeout_seconds
                 self.elapsed = elapsed
                 super().__init__(f"Workflow timed out after {elapsed:.1f}s (limit: {timeout_seconds}s)")
         ```
       - Conditionally (only when `timeout_seconds` is set) add:
         - `import time` at the top imports section
         - `_workflow_start_time = time.monotonic()` before the while-loop
         - At the top of the while-loop body (before the state dispatch), add:
           ```python
           _elapsed = time.monotonic() - _workflow_start_time
           if _elapsed > {{ timeout_seconds }}:
               raise WorkflowTimeoutError({{ timeout_seconds }}, _elapsed)
           ```
       - Use Jinja2 `{% if timeout_seconds %}` guards so none of this code is emitted when timeout is not configured.

    5. **Tests**:
       - In `tests/test_dsl/test_models.py`: Add `TestWorkflowTimeout` class with tests for valid timeout, zero timeout (rejected), negative timeout (rejected), and no timeout (optional).
       - In `tests/test_dsl/test_validator.py`: Add test for extremely large timeout warning.
       - In `tests/test_codegen/test_generator.py`: Add tests confirming timeout code is present when configured and absent when not.
  </implementation>
</feature>
