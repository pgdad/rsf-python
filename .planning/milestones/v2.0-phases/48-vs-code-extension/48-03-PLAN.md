## Goal

Wire the Language Server (plan 48-01) and Graph Preview (plan 48-02) together in the extension entry point, add comprehensive end-to-end integration tests, and prepare the extension for VS Code Marketplace publishing with proper metadata, packaging, and CI workflow.

## Must-Haves (Goal-Backward)

- [ ] Extension activates on opening `workflow.yaml`/`workflow.yml` and starts both LSP and graph preview capabilities
- [ ] Validation errors from LSP appear both as inline diagnostics AND as red borders on graph preview nodes simultaneously
- [ ] Status bar item correctly tracks diagnostic changes from the LSP
- [ ] Graph preview click-to-navigate uses the same state position index as go-to-definition
- [ ] Extension packages into a `.vsix` file with `vsce package` (no errors, correct file listing)
- [ ] `.vsix` includes bundled dependencies and schema — no network requests required at runtime
- [ ] Extension works without any local RSF Python installation
- [ ] README.md with usage instructions and feature screenshots placeholders
- [ ] CHANGELOG.md with initial release entry
- [ ] Integration tests verify full LSP initialization, diagnostic publishing, and definition provider responses

## Tasks

<task id="1" title="Wire components and build integration tests">
### What to build

1. **Update `src/extension.ts`** to integrate all components:
   - Start Language Server (from 48-01 server.ts)
   - Register graph preview command and provider (from 48-02)
   - Connect diagnostic events: when LSP publishes diagnostics, forward error list to graph preview panel
   - Register status bar item that updates from LSP diagnostic events
   - Handle extension deactivation: dispose LSP client, graph preview panel, status bar item
   - Log activation to output channel: "RSF Workflows extension activated"

2. **Update `src/server.ts`** to integrate validation pipeline:
   - On document change (after debounce): run schema validation -> semantic validation -> publish combined diagnostics
   - Send validation results to client for graph preview error highlighting via custom notification
   - Implement `textDocument/definition` using stateNameProvider.getDefinition
   - Implement `textDocument/references` using stateNameProvider.getReferences
   - Implement `textDocument/completion` using stateNameProvider.getCompletions
   - Implement `textDocument/hover` using stateNameProvider.getHover
   - Implement `textDocument/codeAction` using stateNameProvider.getCodeActions
   - Implement `textDocument/documentHighlight` using stateNameProvider.getHighlights

3. **Integration tests in `tests/integration.test.ts`**:
   - Test: Opening a valid workflow.yaml produces 0 diagnostics
   - Test: Opening workflow.yaml with invalid state reference produces diagnostic at correct position
   - Test: Editing workflow.yaml to fix an error clears the corresponding diagnostic
   - Test: Go-to-definition from `Next: "ProcessOrder"` returns correct line/column of ProcessOrder state
   - Test: Find All References for "ProcessOrder" returns all locations where it is referenced
   - Test: Autocomplete in a Next field value position returns all state names
   - Test: Code Action for typo'd state name suggests closest match
   - Test: `yamlToGraph` integration with real validator output — error nodes are correctly identified
   - Test: Schema validation + semantic validation combined — both types of errors appear in diagnostics

   Use fixture files:
   - `tests/fixtures/valid-workflow.yaml` — a correct 4-state workflow
   - `tests/fixtures/invalid-refs.yaml` — workflow with broken state references
   - `tests/fixtures/unreachable.yaml` — workflow with unreachable states
   - `tests/fixtures/parallel-workflow.yaml` — workflow with Parallel branches
   - `tests/fixtures/choice-workflow.yaml` — workflow with Choice state and branches

### Verification
```bash
cd vscode-extension && npm test
```
All integration tests pass. No TypeScript errors.

### Accept/Reject
- **Accept if:** All integration tests pass, extension entry point compiles, all LSP providers wired correctly
- **Reject if:** Integration tests fail, LSP providers not registered, or diagnostic pipeline breaks
</task>

<task id="2" title="Marketplace packaging and CI/CD">
### What to build

1. **Update `package.json`** with complete Marketplace metadata:
   - `publisher`: `rsf-workflows` (placeholder)
   - `repository`: point to rsf-python GitHub repo
   - `icon`: `media/icon.png` (create a simple minimalist workflow graph icon — clean lines, monochrome)
   - `categories`: `["Programming Languages", "Linters", "Visualization"]`
   - `keywords`: `["rsf", "workflow", "state-machine", "yaml", "step-functions"]`
   - `badges`: build status badge
   - `galleryBanner`: theme-appropriate color
   - Marketplace description (short): "YAML schema validation, go-to-definition, and live graph preview for RSF workflow files"

2. **Create `media/icon.png`**: Generate a simple 128x128 SVG-based icon representing a workflow graph (3 connected nodes with directional arrows, monochrome blue on transparent background). Export as PNG. If programmatic generation is complex, create a placeholder and document the requirement.

3. **vscode-extension/README.md** with:
   - Extension name and one-line description
   - Features section with subsections: Schema Validation, Semantic Validation, Go-to-Definition, Graph Preview
   - Screenshots section (placeholder images — document what screenshots are needed)
   - Installation section: "Search for RSF Workflows in VS Code Extensions, or install from Marketplace"
   - Configuration section listing extension settings
   - Known limitations / FAQ

4. **vscode-extension/CHANGELOG.md**:
   - `## 0.1.0` — Initial release with validation, navigation, and graph preview

5. **Build and package script validation**:
   - `npm run build` uses esbuild to bundle extension + server into single files (minimize package size)
   - esbuild config: bundle `src/extension.ts` -> `dist/extension.js` (external: vscode) and `src/server.ts` -> `dist/server.js`
   - `vsce package` produces a `.vsix` file
   - Verify `.vsix` contents: includes dist/, media/, schema, package.json, README, CHANGELOG
   - Verify `.vsix` does NOT include: src/, tests/, node_modules/, tsconfig.json

6. **`.github/workflows/publish.yml`** — CI workflow (reference only, not automated):
   - Trigger: on tag push matching `vscode-extension/v*`
   - Steps: checkout, install deps, build, test, package, publish to Marketplace
   - Uses `@vscode/vsce` for publishing
   - Note: Actual publishing requires Marketplace PAT — document this in README
