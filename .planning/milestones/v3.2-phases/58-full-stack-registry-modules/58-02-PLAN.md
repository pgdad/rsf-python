    Add: A `generate_tfvars()` function (called from BOTH deploy and destroy branches -- per Pitfall 5 from research) that generates terraform.tfvars.json:
    ```bash
    generate_tfvars() {
      jq -n \
        --argjson metadata "$(cat "${METADATA_FILE}")" \
        '{
          workflow_name:         $metadata.workflow_name,
          execution_timeout:     ($metadata.timeout_seconds // 86400),
          dynamodb_tables:       ($metadata.dynamodb_tables // []),
          dlq_enabled:           ($metadata.dlq_enabled // false),
          dlq_queue_name:        $metadata.dlq_queue_name,
          dlq_max_receive_count: ($metadata.dlq_max_receive_count // 3),
          alarms:                [($metadata.alarms // [])[] | {type, threshold, period, evaluation_periods}]
        }' > "${TFVARS_FILE}"
      echo "Generated: ${TFVARS_FILE}"
    }
    ```
    Note the alarms jq expression strips `sns_topic_arn` from each alarm object (Terraform variable type does not include it -- SNS is handled by the module wiring).

    Modify deploy branch: call `generate_tfvars` before terraform apply. Replace `-var` flags with `-var-file="${TFVARS_FILE}"`.

    Modify destroy branch: call `generate_tfvars` before terraform destroy. Replace `-var` flags with `-var-file="${TFVARS_FILE}"`.

    Keep: The echo banner at the top should still print workflow_name. Extract it from METADATA_FILE with a simple `jq -r '.workflow_name'` for display purposes only (not for terraform).
