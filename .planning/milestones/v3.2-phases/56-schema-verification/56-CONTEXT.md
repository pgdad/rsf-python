## Implementation Decisions

### IAM approach
- Let the lambda registry module create and manage the IAM role (module-created, not bring-your-own)
- Compare both RSF's existing Terraform generator IAM output AND the registry module's expected IAM — document the mapping for tutorial side-by-side comparison in Phase 60
- Attach extra durable execution policies to the module-created role via `module.lambda.lambda_role_name`

### Lambda zip path
- Deploy.sh creates a full deployment package: RSF-generated orchestrator code + user handler files together (matches RSF's existing TerraformProvider packaging pattern)
- Zip artifact is gitignored (build artifact, regenerated by deploy.sh on every deploy)

### Verification method
- Confirm durable_config variable names by reading GitHub source at the v8.7.0 tag — no live AWS deploy in this phase
- Check current status of Terraform provider issue #45800 (AllowInvokeLatest) and document findings
- Example directory: `examples/registry-modules-demo/`

### Output artifacts
- Produce both a verification findings document AND a `versions.tf` skeleton with pinned versions in `examples/registry-modules-demo/terraform/`
- Phase 57 builds directly on the versions.tf skeleton

### Claude's Discretion
- Managed policy (AWSLambdaBasicDurableExecutionRolePolicy) vs. inline policy — pick based on verification evidence
- If managed policy is incomplete, whether to supplement with inline or go all-inline — pick based on what verification reveals
- Exact zip path location (build/function.zip vs. terraform/function.zip vs. other) — pick what works best with Terraform's local_existing_package path resolution
- Whether to bundle Python dependencies (pip install) into the zip or rely on Lambda runtime — decide based on Lambda Durable Functions runtime provides
- Which of the other four registry modules (dynamodb-table, sqs, cloudwatch, sns) need upfront schema verification vs. deferring to Phase 58
- Documentation format for verification findings (standalone doc vs. embedded in code)
- Backend configuration placement (in versions.tf skeleton or deferred to Phase 57)
- Terraform resource naming convention (explicit pattern vs. module defaults)

## Existing Code Insights

### Reusable Assets
- `src/rsf/providers/custom.py`: CustomProvider with deploy/teardown dispatch, security-hardened subprocess (shell=False, absolute path validation)
- `src/rsf/providers/metadata.py`: WorkflowMetadata dataclass with all infrastructure fields (workflow_name, dynamodb_tables, alarms, dlq_enabled, etc.)
- `src/rsf/providers/transports.py`: FileTransport writes metadata JSON to RSF_METADATA_FILE — canonical transport for this tutorial
- `src/rsf/terraform/generator.py`: Existing Terraform generator with durable_config and IAM generation — reference for comparison

### Established Patterns
- FileTransport (RSF_METADATA_FILE) is the chosen transport — ArgsTransport cannot handle list/dict metadata fields
- All module versions must be pinned to exact strings (Terraform does not lock module versions in .terraform.lock.hcl)
- Lambda alias convention (never $LATEST) required by Terraform provider issue #45800
- `create_package = false` + `local_existing_package` required to avoid packaging conflict with RSF's zip workflow
- CustomProvider dispatches deploy/destroy via args/teardown_args to a single script

### Integration Points
- `examples/registry-modules-demo/` new directory follows existing example conventions (workflow.yaml, handlers/, tests/, terraform/, README.md, deploy.sh)
- `rsf.toml` configures provider="custom" with absolute path and FileTransport
- deploy.sh reads RSF_METADATA_FILE env var set by FileTransport
