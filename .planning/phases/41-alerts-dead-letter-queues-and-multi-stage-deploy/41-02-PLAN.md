<objective>
Add Lambda dead letter queue support to the RSF DSL, with Pydantic validation, semantic checks, and Terraform generation including an SQS DLQ wired to the Lambda function.

Purpose: Users need to configure dead letter queues for their workflow Lambda functions so failed invocations (after all retries are exhausted) are captured for debugging. `rsf generate` produces the SQS DLQ Terraform resources, wires them to the Lambda function's dead_letter_config, and grants the necessary IAM permissions.

Output: DeadLetterQueueConfig Pydantic model, semantic validation, dlq.tf.j2 Terraform template, updated main.tf.j2 with dead_letter_config, updated IAM template with SQS DLQ permissions, and comprehensive TDD tests.
</objective>

    Note: The redrive policy is configured on the SOURCE queue (not the DLQ itself). For Lambda DLQ, the redrive is handled by the Lambda dead_letter_config, not an SQS redrive policy. The max_receive_count maps to Lambda's retry behavior — Lambda will retry the configured number of times and then send the event to the DLQ. However, Lambda's native DLQ (via dead_letter_config) doesn't use maxReceiveCount directly — it sends failed async invocations to the DLQ after Lambda's internal retry policy (currently 2 retries for async). For event source mappings (SQS triggers), maxReceiveCount IS used. Since this is for the Lambda function itself (async invocation failures), use Lambda's `dead_letter_config` block which simply specifies the target ARN. The `max_receive_count` field can be documented as informational for event source mappings.
