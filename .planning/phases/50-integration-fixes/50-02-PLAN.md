**5. Integrate chaos check into _call_handler_with_retry:**

At the top of `_call_handler_with_retry`, BEFORE loading the handler, check for chaos injection:
```python
def _call_handler_with_retry(self, name: str, state: TaskState, data: Any) -> Any:
    """Call a handler with retry logic."""
    # Check for chaos injection BEFORE calling real handler
    if self.chaos_fixture is not None:
        failure = self.chaos_fixture._should_trigger(name)
        if failure is not None:
            self.chaos_fixture._trigger_failure(failure, data)

    handler_fn = _load_handler(name, self.workflow_dir)
    # ... rest of retry logic unchanged ...
```

**IMPORTANT:** The chaos check must happen BEFORE the retry loop, not inside it. A chaos-injected failure should bypass the handler entirely and raise immediately (not be caught by retry logic). This matches the ChaosFixture API design where injected failures simulate infrastructure-level failures that bypass application retry.

Wait -- on reflection, chaos injection should interact WITH retry logic to test retry behavior. The chaos injection should happen where `handler_fn(data)` is called inside the retry loop, so that retry policies can catch and retry chaos failures. This is the more useful behavior for testing retry logic.

**REVISED approach for step 5:** Wrap the handler function to check chaos first:

```python
def _call_handler_with_retry(self, name: str, state: TaskState, data: Any) -> Any:
    """Call a handler with retry logic."""
    handler_fn = _load_handler(name, self.workflow_dir)

    # Wrap handler with chaos injection if active
    if self.chaos_fixture is not None:
        original_handler = handler_fn
        def chaos_wrapped_handler(d):
            failure = self.chaos_fixture._should_trigger(name)
            if failure is not None:
                return self.chaos_fixture._trigger_failure(failure, d)
            return original_handler(d)
        handler_fn = chaos_wrapped_handler

    retry_policies = state.retry or []
    # ... rest unchanged, uses handler_fn which is now chaos-wrapped ...
```

This way:
- Chaos failures participate in retry/catch logic (testable!)
- count-based chaos (e.g., fail 2 times then succeed) works with retry
- Non-injected states call the real handler normally
