## Summary

Phase 56 is a research-and-document phase: it produces verified facts that all downstream Terraform code (Phases 57-60) must rest on. The three critical unknowns as of project research were: (1) exact durable_config input variable names in terraform-aws-modules/lambda v8.7.0, (2) current status of Terraform provider issue #45800, and (3) the correct IAM approach given potential mismatch between RSF's existing inline policy and the AWS managed policy. All three are now resolved with HIGH confidence from live primary sources.

**durable_config variables confirmed (HIGH confidence, from live v8.7.0 tag on GitHub):** The module exposes exactly two variables: `durable_config_execution_timeout` (number, null default) and `durable_config_retention_period` (number, null default). The dynamic block activates only when `durable_config_execution_timeout != null`. Setting this variable to any non-null value is the activation gate — if it is null, the entire `durable_config` block is silently omitted from the Lambda resource, which would produce a non-durable function with no error.

**Issue #45800 status (MEDIUM confidence, from GitHub issue page):** Still open as of January 7, 2026 (last activity date). `AllowInvokeLatest` has not been added to the Terraform AWS provider because the AWS SDK for Go v2 has not implemented it yet. The "always use alias" convention must be established in this phase and followed throughout Phases 57-60.

**IAM approach decision (HIGH confidence from official managed policy reference):** The managed policy `AWSLambdaBasicDurableExecutionRolePolicy` contains exactly: `lambda:CheckpointDurableExecution`, `lambda:GetDurableExecutionState`, plus CloudWatch Logs permissions. RSF's existing inline policy uses different action names: `lambda:GetDurableExecution` (not `GetDurableExecutionState`) and includes `lambda:ListDurableExecutionsByFunction` and `lambda:InvokeFunction` — neither of which is in the managed policy. The recommended approach is: attach the managed policy via `attach_policies` + `number_of_policies`, then add an inline policy for self-invoke and list operations via `attach_policy_json`.

**Primary recommendation:** Use managed policy `AWSLambdaBasicDurableExecutionRolePolicy` as the base (covers checkpoint + state retrieval + logs), supplemented by an inline policy for `lambda:InvokeFunction` (self-invoke) and `lambda:ListDurableExecutionsByFunction`. This approach is documented alongside RSF's existing pure-inline approach for the Phase 60 tutorial comparison.

### Pattern 1: durable_config Activation Gate

**What:** The lambda module activates durable_config only when `durable_config_execution_timeout != null`.
**When to use:** Always set `durable_config_execution_timeout` to a non-null value when deploying a durable Lambda. Omitting it or setting it to null produces a non-durable function silently.

```hcl
# Source: https://raw.githubusercontent.com/terraform-aws-modules/terraform-aws-lambda/v8.7.0/variables.tf
# Variable declaration (confirmed from live v8.7.0 tag):
variable "durable_config_execution_timeout" {
  type    = number
  default = null
  description = "Maximum execution time in seconds for the durable function. Valid values between 1 and 31622400 (366 days)."
}

variable "durable_config_retention_period" {
  type    = number
  default = null
  description = "Number of days to retain the function's execution state. Valid values between 1 and 90. Defaults to 14 if durable_config is enabled."
}

# Dynamic block from module main.tf (confirmed from live v8.7.0 tag):
dynamic "durable_config" {
  for_each = var.durable_config_execution_timeout != null ? [true] : []
  content {
    execution_timeout = var.durable_config_execution_timeout
    retention_period  = var.durable_config_retention_period
  }
}
```

**Usage in tutorial Terraform:**
```hcl
module "lambda" {
  source  = "terraform-aws-modules/lambda/aws"
  version = "8.7.0"  # exact — not ~> 8.7

  function_name = var.workflow_name
  handler       = "generated.orchestrator.lambda_handler"
  runtime       = "python3.13"

  # RSF manages packaging — disable module's built-in packager
  create_package         = false
  local_existing_package = "${path.module}/../build/function.zip"
  ignore_source_code_hash = true

  # durable_config activation — MUST be non-null to enable
  durable_config_execution_timeout = var.execution_timeout   # e.g. 86400 for 24h
  durable_config_retention_period  = 14                      # days

  # IAM role — module creates it; we supplement with durable permissions
  create_role                    = true
  attach_cloudwatch_logs_policy  = true
  attach_policies                = true
  number_of_policies             = 1
  policies = [
    "arn:aws:iam::aws:policy/service-role/AWSLambdaBasicDurableExecutionRolePolicy"
  ]
  attach_policy_json = true
  policy_json = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Sid    = "DurableSelfInvokeAndList"
      Effect = "Allow"
      Action = [
        "lambda:InvokeFunction",
        "lambda:ListDurableExecutionsByFunction"
      ]
      Resource = "*"  # scoped to self via alias ARN at invocation time
    }]
  })
}
```

### Pattern 2: Lambda Alias — Always Use Alias, Never $LATEST

**What:** Create a Lambda alias and use its ARN for all invocations. Never invoke via `$LATEST` while Terraform provider issue #45800 is unresolved.
**Why:** Terraform AWS provider does not yet support `AllowInvokeLatest` in the durable_config block (issue #45800, open as of Jan 7, 2026). Default behavior when AllowInvokeLatest is false may cause invocation failures via unqualified ARN or `$LATEST`.
**Official AWS docs pattern:** The AWS IaC guide publishes a version and creates an alias named `"prod"`. Tutorial will use `"live"` as the canonical alias name.

```hcl
# Source: https://docs.aws.amazon.com/lambda/latest/dg/durable-getting-started-iac.html
resource "aws_lambda_alias" "live" {
  name             = "live"
  function_name    = module.lambda.lambda_function_name
  function_version = module.lambda.lambda_function_version
}

output "alias_arn" {
  description = "Use this ARN for all durable function invocations (never $LATEST)"
  value       = aws_lambda_alias.live.arn
}
```

### Pattern 3: IAM Hybrid Approach (Managed + Inline Supplement)

**What:** Attach `AWSLambdaBasicDurableExecutionRolePolicy` (managed) via `attach_policies`, then add an inline policy via `attach_policy_json` for operations not covered by the managed policy.
**Why:** The managed policy covers checkpoint and state retrieval operations, but does NOT include self-invoke (`lambda:InvokeFunction`) or list operations (`lambda:ListDurableExecutionsByFunction`) — both required for RSF's durable orchestration pattern.

**Managed policy actions (confirmed from AWS managed policy reference):**
- `lambda:CheckpointDurableExecution`
- `lambda:GetDurableExecutionState`
- `logs:CreateLogGroup`, `logs:CreateLogStream`, `logs:PutLogEvents`

**RSF inline policy EXISTING actions (from iam.tf.j2):**
- `lambda:CheckpointDurableExecution` — matches managed policy (same name, confirmed)
- `lambda:GetDurableExecution` — DIFFERENT from managed policy's `GetDurableExecutionState`
- `lambda:ListDurableExecutionsByFunction` — NOT in managed policy (must be inline)
- `lambda:InvokeFunction` — NOT in managed policy (must be inline for self-invoke)

**Action name discrepancy to document in Phase 60 comparison:**
The managed policy uses `GetDurableExecutionState`; RSF's existing inline policy uses `GetDurableExecution`. Both API endpoints exist in AWS (verified from AWS docs links). The managed policy covers the SDK's state retrieval path; `GetDurableExecution` is the user-facing describe API. The tutorial IAM must include both to cover all SDK call paths used by the durable runtime.

### Pattern 4: Lambda Zip Path Convention

**Decision:** `build/function.zip` in the example root directory (not inside terraform/).

**Rationale:**
- `local_existing_package` accepts an absolute path or path relative to the module directory (`path.module`)
- From `terraform/`, the path `${path.module}/../build/function.zip` resolves cleanly
- The `build/` directory is gitignored (alongside `.terraform/`, `*.tfstate`)
- `deploy.sh` creates `build/function.zip` by zipping: `src/generated/` (RSF orchestrator) + `handlers/` (user business logic) together — matches RSF's TerraformProvider packaging which uses `archive_file` over both directories combined
- Location mirrors the existing `examples/order-processing/terraform/order-processing.zip` pattern but moves the artifact out of `terraform/` to a dedicated `build/` directory

**deploy.sh zip step:**
```bash
# Source: established pattern from CONTEXT.md decision
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
mkdir -p "${SCRIPT_DIR}/build"
cd "${SCRIPT_DIR}"
zip -r "${SCRIPT_DIR}/build/function.zip" src/generated/ handlers/ -x "**/__pycache__/*" "**/*.pyc"
```

**Terraform reference:**
```hcl
local_existing_package = "${path.module}/../build/function.zip"
```

### Pattern 5: Module Version Exact Pinning

**What:** Use exact version strings (`version = "8.7.0"`) not range constraints (`~> 8.7`).
**Why:** Terraform does not include module versions in `.terraform.lock.hcl`. A `~> 8.7` constraint silently upgrades on `terraform init` as new versions are released. An `8.8.0` release could change output names or variable behavior, breaking every tutorial reader who runs `terraform init` after the release.

```hcl
# versions.tf skeleton — Phase 56 deliverable
# Source: confirmed from all module GitHub releases pages
terraform {
  required_version = ">= 1.5.7"  # Required by sqs v5.x and sns v7.x modules

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = ">= 6.25.0"  # RSF durable_config requirement (provider v6.25.0, Dec 4 2025)
    }
    archive = {
      source  = "hashicorp/archive"
      version = ">= 2.7.1"
    }
  }
}

# Registry module version pins — exact strings, not ranges
# Rationale: Terraform does not lock module versions in .terraform.lock.hcl.
# Using ~> would silently upgrade on terraform init and may break documented behavior.
locals {
  # These pins are read by Phase 57-58 module source blocks.
  # Do not change without reviewing the module's CHANGELOG.
  lambda_module_version    = "8.7.0"   # First version with durable_config support
  dynamodb_module_version  = "5.5.0"   # Verified Jan 8, 2026
  sqs_module_version       = "5.2.1"   # Verified Jan 21, 2026
  cloudwatch_module_version = "5.7.2"  # Verified Oct 21, 2025
  sns_module_version       = "7.1.0"   # Verified Jan 8, 2026
}
```

Note: Terraform `locals` cannot be used in `module.source` or `module.version` blocks — these version strings are documented as `locals` for reference only. Actual `version = "8.7.0"` strings appear in each module block directly.

### Anti-Patterns to Avoid

- **`durable_config_execution_timeout = null` (or omitted):** The dynamic block gate means the entire durable_config is silently dropped. Always set a numeric value.
- **`version = "~> 8.7"` on registry modules:** Silently upgrades on `terraform init`. Use exact strings.
- **`create_package = true` (default):** Conflicts with RSF's zip creation. Always set `create_package = false`.
- **Invoking via `$LATEST` or unqualified ARN:** Use alias ARN while issue #45800 is unresolved.
- **Omitting `attach_policies = true` and `number_of_policies`:** The `policies` variable is only processed when `attach_policies = true`. Setting `policies = [...]` without `attach_policies = true` silently attaches nothing.
- **Using `GetDurableExecution` only:** The managed policy uses `GetDurableExecutionState`. The RSF SDK may call both APIs. Include both action names across the managed + inline split.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Lambda IAM role creation | Custom `aws_iam_role` + trust policy + CW Logs attachment | lambda module `create_role = true` (default) | Module handles trust policy, CloudWatch Logs, dead letter, VPC, X-Ray automatically. Hand-rolled role misses edge cases. |
| Lambda packaging | `data.archive_file` source_path inside module | `create_package = false` + `local_existing_package` | RSF already creates the zip. Two packaging systems = hash mismatches and non-deterministic deploys. |
| DynamoDB table per workflow | Multiple `aws_dynamodb_table` resources | `terraform-aws-modules/dynamodb-table/aws` with `for_each` | Module handles billing mode, attribute definitions, GSI, and consistent naming. |
| SQS DLQ | Raw `aws_sqs_queue` | `terraform-aws-modules/sqs/aws` with `count` gate | Module handles encryption, policy, redrive policy, and output naming consistently. |
| CloudWatch metric alarms | Raw `aws_cloudwatch_metric_alarm` | `terraform-aws-modules/cloudwatch/aws//modules/metric-alarm` | Module handles comparison operators, treat_missing_data defaults. Must use `//modules/metric-alarm` submodule path. |
| Version constraint enforcement | Comment in code | Exact `version = "8.7.0"` string in every module block | Terraform will not enforce pinning without exact version in source block. |

**Key insight:** The lambda module replaces RSF's entire `main.tf` + `iam.tf` + `cloudwatch.tf` combination with a single module block. The primary teaching point is that `durable_config` and IAM are no longer separate files — but durable-specific permissions (self-invoke, list) still require manual supplementation.

## Common Pitfalls

### Pitfall 1: durable_config Silently Dropped When Timeout Is Null

**What goes wrong:** Setting `durable_config_execution_timeout = var.timeout_seconds` where `var.timeout_seconds` defaults to `null` (WorkflowMetadata field) results in the `dynamic "durable_config"` block being skipped entirely. Lambda is created without durable execution enabled. `terraform apply` succeeds silently.
**Why it happens:** The dynamic block gate is `var.durable_config_execution_timeout != null`. A null variable is the module default. Developers pass metadata values directly without ensuring non-null.
**How to avoid:** Always pass a concrete integer: `durable_config_execution_timeout = coalesce(var.execution_timeout, 86400)`. Document this in the Terraform variables.tf as required with no default.
**Warning signs:** `aws lambda get-function --query 'Configuration.DurableConfig'` returns null after apply.

### Pitfall 2: AllowInvokeLatest Not Configurable (Issue #45800 Still Open)

**What goes wrong:** Invoking a durable function via unqualified ARN or `$LATEST` may fail if `AllowInvokeLatest` defaults to false at the AWS API level. The Terraform provider cannot set this field.
**Why it happens:** AWS SDK for Go v2 had not implemented `AllowInvokeLatest` as of the Terraform provider's v6.25.0 release. Issue #45800 is still open (last activity Jan 7, 2026).
**How to avoid:** Always create a Lambda alias and invoke via alias ARN. Document explicitly: "All invocations in this tutorial use the `live` alias ARN, not `$LATEST`."
**Warning signs:** `aws lambda invoke --function-name arn:...:$LATEST` fails with ResourceConflictException or similar durable-specific error.

### Pitfall 3: IAM Action Name Mismatch Between RSF Generator and Managed Policy

**What goes wrong:** RSF's iam.tf.j2 uses `lambda:GetDurableExecution`. The managed policy `AWSLambdaBasicDurableExecutionRolePolicy` uses `lambda:GetDurableExecutionState`. These are different APIs (describe vs. replay-state). Using only one may cause durable SDK calls to fail with `AccessDeniedException`.
**Why it happens:** Lambda Durable Functions launched at re:Invent 2025. IAM action names stabilized over Dec 2025 — Jan 2026. RSF's generator predates the managed policy publication.
**How to avoid:** Use both: managed policy covers `GetDurableExecutionState` (replay path); add inline policy for `GetDurableExecution` (if needed by RSF SDK). Document the distinction. Run a real durable invocation to completion (not just `terraform apply`) before declaring IAM correct.
**Warning signs:** `terraform apply` succeeds but first durable invocation fails with `AccessDeniedException` on a durable API action.

### Pitfall 4: `attach_policies` Requires BOTH the Bool and the Count

**What goes wrong:** Setting `policies = ["arn:..."]` without `attach_policies = true` and `number_of_policies = 1` silently attaches nothing. The module only processes the `policies` list when `attach_policies` is true AND `number_of_policies` matches the list length.
**Why it happens:** The module uses `count = local.create_role && var.attach_policies ? var.number_of_policies : 0` on the policy attachment resource. Without the bool, count is 0.
**How to avoid:** Always set all three: `attach_policies = true`, `number_of_policies = N`, and `policies = [list of N ARNs]`.

### Pitfall 5: Cloudwatch Module Root Has No Configuration

**What goes wrong:** Using `source = "terraform-aws-modules/cloudwatch/aws"` (root module) fails with an error — the root module has no configuration. Must use the submodule path.
**How to avoid:** Always use `source = "terraform-aws-modules/cloudwatch/aws//modules/metric-alarm"` with the double-slash submodule syntax.

## IAM Approach Decision

**Decision: Use managed policy + inline supplement (not all-inline).**

**Evidence:**
1. `AWSLambdaBasicDurableExecutionRolePolicy` is the official AWS-recommended managed policy for Lambda Durable Functions. Confirmed from AWS managed policy reference guide (v3, updated Feb 12, 2026).
2. The managed policy contains exactly: `lambda:CheckpointDurableExecution`, `lambda:GetDurableExecutionState`, plus CloudWatch Logs permissions.
3. Self-invoke (`lambda:InvokeFunction`) and `lambda:ListDurableExecutionsByFunction` are NOT in the managed policy. The RSF durable runtime requires both. These must be inline.
4. `lambda:GetDurableExecution` (user-facing describe API, different from `GetDurableExecutionState` which is the replay-state API) is used in RSF's existing generator. Including it as a supplemental inline action is safe (additive permission, not harmful).

**Action name mapping for Phase 60 tutorial comparison:**

| Source | Action | Managed Policy | RSF Inline (existing) | Tutorial IAM |
|--------|--------|---------------|----------------------|--------------|
| AWS managed policy | `lambda:CheckpointDurableExecution` | YES | YES | via managed policy |
| AWS managed policy | `lambda:GetDurableExecutionState` | YES | NO | via managed policy |
| RSF existing | `lambda:GetDurableExecution` | NO | YES | via inline supplement |
| RSF existing | `lambda:ListDurableExecutionsByFunction` | NO | YES | via inline supplement |
| RSF existing | `lambda:InvokeFunction` | NO | YES | via inline supplement |
| AWS managed policy | `logs:CreateLogGroup/Stream/PutLogEvents` | YES | YES | via managed policy (module handles via `attach_cloudwatch_logs_policy`) |

## Lambda Zip Path Decision

**Decision: `build/function.zip` in the example root, referenced as `${path.module}/../build/function.zip` from `terraform/`.**

**Rationale:**
- Keeps build artifacts outside the Terraform working directory (avoids Terraform seeing the zip as a module asset to include in its own packaging)
- `${path.module}/../build/function.zip` is a clean relative path from `terraform/` to `build/`
- `build/` directory is gitignored alongside `.terraform/`, `*.tfstate`, `*.tfstate.backup`
- `deploy.sh` zips: `src/generated/` (RSF orchestrator output) + `handlers/` (user code) into `build/function.zip` — mirrors the RSF TerraformProvider's `data.archive_file` which zips the `src/` directory

**Python dependencies decision:** Do NOT bundle pip packages. Lambda Durable Functions runtime runs Python 3.13. RSF's orchestrator and user handlers have no external pip dependencies beyond the RSF SDK itself, which is not needed at runtime (RSF generates standalone code). The zip contains only the generated orchestrator + user handler Python files.

## Open Questions

1. **`lambda:GetDurableExecution` vs `lambda:GetDurableExecutionState` at runtime**
   - What we know: The managed policy uses `GetDurableExecutionState` (replay path). RSF's existing generator grants `GetDurableExecution` (describe API). Both API endpoints exist.
   - What's unclear: Whether the RSF durable SDK calls `GetDurableExecution` at runtime (user-facing), `GetDurableExecutionState` (internal replay), or both.
   - Recommendation: Include both in the inline supplement to be safe. Validate with a real durable invocation in Phase 57.

2. **`AllowInvokeLatest` resolution timing**
   - What we know: Issue #45800 is open. Last activity Jan 7, 2026. Depends on AWS SDK for Go v2.
   - What's unclear: Whether it will be resolved before tutorial publication.
   - Recommendation: Use alias convention throughout and add a note: "If issue #45800 is resolved, `AllowInvokeLatest = true` can be added to the `durable_config` block to enable `$LATEST` invocations."

3. **`AWSLambdaBasicDurableExecutionRolePolicy` regional availability**
   - What we know: Policy confirmed to exist (managed policy reference guide shows v3, Feb 12, 2026). Regional rollout timing referenced in Pulumi issue #6100 as potentially incomplete in early 2026.
   - What's unclear: Whether the policy is available in us-east-2 (project target region) as of March 2026.
   - Recommendation: Phase 57 MUST verify with `aws iam get-policy --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicDurableExecutionRolePolicy` before writing Terraform code that attaches it. Fallback: all-inline policy matching RSF's existing iam.tf.j2 approach.
