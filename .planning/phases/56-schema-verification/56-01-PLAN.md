```hcl
# examples/registry-modules-demo/terraform/versions.tf
# Produced by Phase 56 schema verification.
# All module versions pinned to exact strings — Terraform does not lock
# module versions in .terraform.lock.hcl. Range constraints (~>) would
# silently upgrade on terraform init and may break documented behavior.

terraform {
  required_version = ">= 1.5.7"  # Required by sqs v5.x and sns v7.x

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = ">= 6.25.0"  # durable_config block (provider v6.25.0, Dec 4 2025)
    }
    archive = {
      source  = "hashicorp/archive"
      version = ">= 2.7.1"
    }
  }
}

provider "aws" {
  region = var.aws_region
}

# Registry module version pins — exact strings, not ranges.
# These version strings appear in each module's source block in their
# respective .tf files (main.tf, dynamodb.tf, sqs.tf, alarms.tf).
# Terraform locals cannot be used in module source/version blocks,
# so these serve as authoritative reference documentation.
#
# Module                                      | Version | Released   | Notes
# --------------------------------------------|---------|------------|-------------------------------
# terraform-aws-modules/lambda/aws            | 8.7.0   | 2026-02-18 | First version with durable_config support
# terraform-aws-modules/dynamodb-table/aws    | 5.5.0   | 2026-01-08 | Verified from GitHub releases
# terraform-aws-modules/sqs/aws              | 5.2.1   | 2026-01-21 | Requires Terraform >= 1.5.7
# terraform-aws-modules/cloudwatch/aws        | 5.7.2   | 2025-10-21 | Must use //modules/metric-alarm submodule
# terraform-aws-modules/sns/aws              | 7.1.0   | 2026-01-08 | Requires AWS provider >= 6.9
```

Key points:
- `required_version = ">= 1.5.7"` (not `>= 1.0` like the existing order-processing example) because sqs v5.x and sns v7.x modules require it.
- `archive` version bumped to `>= 2.7.1` to match RSF's current constraint.
- Module versions are documented as a reference table in comments (locals cannot be used in module source blocks).
- The `provider "aws"` block uses `var.aws_region` following the existing RSF convention.
- Do NOT include a backend block — deferred to Phase 57 per Claude's Discretion decision.

**Section 1: durable_config Variable Names (Confirmed)**
- Exact variable names: `durable_config_execution_timeout` (number, null default) and `durable_config_retention_period` (number, null default)
- Source: `https://raw.githubusercontent.com/terraform-aws-modules/terraform-aws-lambda/v8.7.0/variables.tf`
- Activation gate: dynamic block uses `var.durable_config_execution_timeout != null` — null = silently no durable config
- Include the exact HCL snippet showing module usage:
  ```hcl
  durable_config_execution_timeout = 86400  # MUST be non-null
  durable_config_retention_period  = 14     # days, optional
  ```
- CRITICAL WARNING: Always use `coalesce(var.execution_timeout, 86400)` to prevent null propagation from WorkflowMetadata

**Section 2: Lambda Alias Convention**
- Convention: Always create a Lambda alias named `"live"` and invoke via alias ARN
- Never use `$LATEST` or unqualified ARN
- Rationale: Terraform provider issue #45800 (AllowInvokeLatest) still open as of Jan 7, 2026
- AWS SDK for Go v2 has not implemented AllowInvokeLatest
- Include the alias HCL pattern:
  ```hcl
  resource "aws_lambda_alias" "live" {
    name             = "live"
    function_name    = module.lambda.lambda_function_name
    function_version = module.lambda.lambda_function_version
  }
  ```
- Note: If issue #45800 is resolved before tutorial publication, `AllowInvokeLatest = true` can be added to the durable_config block

**Section 3: IAM Approach Decision**
- Decision: Managed policy + inline supplement (hybrid approach)
- Managed policy: `AWSLambdaBasicDurableExecutionRolePolicy` covers `CheckpointDurableExecution`, `GetDurableExecutionState`, CW Logs
- Inline supplement: `InvokeFunction` (self-invoke), `ListDurableExecutionsByFunction`, `GetDurableExecution` (describe API)
- Include the full action name mapping table from 56-RESEARCH.md (6 rows, columns: Source, Action, Managed Policy, RSF Inline Existing, Tutorial IAM)
- Module attachment pattern: `attach_policies = true`, `number_of_policies = 1`, `policies = [ARN]`, `attach_policy_json = true`
- WARNING: Setting `policies = [...]` without `attach_policies = true` silently attaches nothing
- WARNING: Regional availability of the managed policy must be verified in Phase 57 with `aws iam get-policy`
- Fallback: If managed policy unavailable in us-east-2, use all-inline matching RSF's existing iam.tf.j2

**Section 4: Registry Module Version Pins**
- Table with all 5 modules: name, exact version, release date, purpose, notes
- lambda 8.7.0 (Feb 18, 2026) — first with durable_config
- dynamodb-table 5.5.0 (Jan 8, 2026)
- sqs 5.2.1 (Jan 21, 2026) — requires TF >= 1.5.7
- cloudwatch 5.7.2 (Oct 21, 2025) — must use //modules/metric-alarm submodule path
- sns 7.1.0 (Jan 8, 2026) — requires AWS provider >= 6.9
- Rationale for exact pins: Terraform does not include module versions in .terraform.lock.hcl

**Section 5: Lambda Zip Path Convention**
- Path: `build/function.zip` in example root
- Terraform reference: `${path.module}/../build/function.zip`
- deploy.sh creates zip from: `src/generated/` (RSF orchestrator) + `handlers/` (user code)
- No pip dependencies bundled — Lambda Durable Functions runtime provides Python 3.13 stdlib
- `build/` directory is gitignored
- Module setting: `create_package = false`, `local_existing_package = "${path.module}/../build/function.zip"`, `ignore_source_code_hash = true`

**Section 6: Anti-Patterns (Quick Reference)**
Bulleted list of the 6 anti-patterns from research:
1. durable_config_execution_timeout = null (silently drops durable config)
2. version = "~> 8.7" on registry modules (silent upgrades)
3. create_package = true (conflicts with RSF zip)
4. Invoking via $LATEST (issue #45800)
5. Omitting attach_policies = true with policies list (silently attaches nothing)
6. Using cloudwatch root module instead of //modules/metric-alarm submodule

**Section 7: Open Questions for Phase 57**
1. GetDurableExecution vs GetDurableExecutionState runtime usage — include both, validate with live invocation
2. AllowInvokeLatest resolution timing — use alias convention throughout
3. AWSLambdaBasicDurableExecutionRolePolicy regional availability in us-east-2 — must verify before writing Terraform

This document should be self-contained — a Phase 57 executor reading only this file plus versions.tf should have everything needed to write correct Terraform without re-reading the research.
