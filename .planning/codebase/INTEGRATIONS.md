# External Integrations

**Analysis Date:** 2026-03-11

## APIs & External Services

**AWS Lambda (Primary):**
- Service: AWS Lambda function invocation, durable execution APIs
  - SDK/Client: boto3 (`import boto3`)
  - Used in: `src/rsf/inspect/client.py`, `src/rsf/providers/cdk.py`, deployment code
  - Methods: `list_durable_executions_by_function`, `get_durable_execution`, `invoke`
  - Auth: AWS credentials via boto3 session (standard IAM role assumption)

**AWS IAM:**
- Service: IAM role/policy generation and inspection
  - SDK/Client: boto3 client
  - Used in: `src/rsf/terraform/generator.py` (derives IAM statements for Lambda execution)
  - Function: `derive_iam_statements()` generates JSON policy documents

**AWS CloudWatch Logs:**
- Service: Execution logs and diagnostics
  - SDK/Client: boto3 (referenced indirectly)
  - Used in: `src/rsf/cli/logs_cmd.py`

**AWS EC2 / VPC (Template-based):**
- Service: Infrastructure for Lambda deployments (optional)
  - Generated by: `src/rsf/terraform/generator.py` and `src/rsf/cdk/generator.py`
  - VPC integration for Lambda functions (when templates specify it)

## Data Storage

**Databases:**
- DynamoDB
  - Connection: Via boto3 resource
  - ORM/Client: boto3 dynamodb resource
  - Used in: Example templates (`src/rsf/cli/templates/api-gateway-crud/`)
  - Pattern: Handlers use `dynamodb.Table()` for CRUD operations

**File Storage:**
- S3 (Simple Storage Service)
  - Connection: Via boto3 client
  - Used in: Example templates (`src/rsf/cli/templates/s3-event-pipeline/`)
  - Pattern: S3 events trigger Lambda handlers

**Local Filesystem:**
- Project configuration: `rsf.toml` loaded from workflow directory
- Workflow YAML files: Local file-based workflow definitions
- Output artifacts: Generated Terraform/CDK code written to `--output-dir`

**Caching:**
- None detected. In-memory state for FastAPI applications (Uvicorn worker state).

## Authentication & Identity

**Auth Provider:**
- AWS IAM (implicit)
  - Implementation: boto3 credential chain (env vars, instance metadata, shared credentials file)
  - No explicit auth server or OAuth provider
  - Lambda function assumes execution role via AWS STS (Security Token Service)

**No Custom Auth:**
- No API key requirements in RSF CLI or servers
- No OAuth/OIDC integration
- Editor and Inspector servers (FastAPI) have no authentication guards

## Environment Configuration

**Required env vars:**
- AWS_REGION or boto3 default region (for Lambda client)
- AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY (or EC2 instance profile)
- SSH_AUTH_SOCK (for CDK/npm operations, if SSH key required)

**Optional env vars:**
- PYTHONPATH (set to `src` in pytest config for imports)
- RSF_STAGE (for deployment staging, passed to infrastructure providers)

**Secrets location:**
- AWS credentials: Standard AWS SDK credential chain
  - Environment variables: `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`
  - Shared credentials file: `~/.aws/credentials`
  - Instance metadata (EC2 IAM role)
- No `.env` file mechanism in codebase

**Configuration Files:**
- `rsf.toml` - Project-level infrastructure configuration (optional, in workflow directory)
- `workflow.yaml` - Workflow definition with optional `infrastructure:` block (overrides rsf.toml)

## Rate Limiting & Throttling

**Lambda Control-Plane Limits:**
- Token-bucket rate limiter in `src/rsf/inspect/client.py`
  - Rate: 12 requests/second (default)
  - Capacity: 12 tokens
  - Purpose: Stay under Lambda's 15 req/s control-plane limit
  - Applied to: `list_durable_executions_by_function`, `get_durable_execution`, `invoke`
